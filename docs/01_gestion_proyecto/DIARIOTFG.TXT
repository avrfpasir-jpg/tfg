# Fase 1: Infraestructura de Red (SENTINEL-VPC)

En esta primera fase se ha desplegado la arquitectura de red básica sobre la cual se ejecutarán los servicios del proyecto. Se ha optado por un diseño de **Virtual Private Cloud (VPC)** segmentada para garantizar la seguridad y el control del tráfico.

## 1. Configuración de la VPC

Se ha creado un entorno de red aislado con las siguientes especificaciones:

- **Nombre:** `SENTINEL-VPC`
    
- **Bloque CIDR IPv4:** `10.0.0.0/16` (65,536 direcciones IP disponibles).
    

## 2. Segmentación de Subredes

Para separar los niveles de acceso, se han definido dos subredes en la misma zona de disponibilidad:

|**Subred**|**Rango CIDR**|**Tipo**|**Propósito**|
|---|---|---|---|
|`Subred-Publica`|`10.0.1.0/24`|Pública|Alojar el Bastion Host y servicios con cara al exterior.|
|`Subred-Privada`|`10.0.2.0/24`|Privada|Alojar la lógica de negocio y bases de datos sin acceso directo desde internet.|

## 3. Conectividad Exterior (Internet Gateway)

Para dotar de salida a internet a la red, se han configurado los siguientes componentes:

- **Gateway:** Creación de `SENTINEL-IGW` y asociación directa a `SENTINEL-VPC`.
    
- **Enrutamiento:** Se ha modificado la tabla de rutas principal para dirigir todo el tráfico externo (`0.0.0.0/0`) hacia el **Internet Gateway** (Puerta de enlace de Internet).
    
- **Asociación:** Se ha vinculado la `Subred-Publica` a esta tabla de rutas, habilitando así la comunicación bidireccional con el exterior para esta zona.

# Fase 2: Seguridad Perimetral y Control de Acceso (Security Groups)

En esta fase se han diseñado e implementado los mecanismos de seguridad a nivel de red para proteger los activos del proyecto **SENTINEL**. Se ha utilizado un modelo de seguridad por capas mediante el uso de **Security Groups (SG)**, que actúan como firewalls con estado (_stateful_).

## 1. Segmentación de Grupos de Seguridad

Se han definido dos grupos diferenciados para separar las responsabilidades de red:

|**Nombre del Grupo**|**Función**|**Ámbito**|
|---|---|---|
|`SG-WEB`|Servidor de Aplicaciones / Bastion|Público|
|`SG-DB`|Almacenamiento y Datos|Privado|

## 2. Configuración de Reglas de Entrada (Inbound Rules)

### 2.1. Nivel de Aplicación (`SG-WEB`)

Se han habilitado los puertos necesarios para la gestión y visualización del sistema:

- **SSH (Puerto 22):** Permitido desde `0.0.0.0/0` para facilitar el acceso administrativo remoto desde cualquier red durante la fase de desarrollo.
    
- **HTTP (Puerto 80):** Permitido desde `0.0.0.0/0` para permitir el acceso de usuarios a la interfaz de monitorización de SENTINEL.
    

### 2.2. Nivel de Datos (`SG-DB`)

Se ha aplicado una configuración restrictiva mediante **Seguridad por Referencia**:

- **MySQL/Aurora (Puerto 3306):** El origen se ha limitado exclusivamente al ID del grupo `SG-WEB`.
    
- **Justificación técnica:** Esta configuración garantiza que la base de datos sea invisible para todo internet y solo acepte peticiones que provengan de instancias autorizadas dentro del grupo web, reduciendo drásticamente la superficie de ataque.
    

## 3. Reglas de Salida (Outbound Rules)

Para ambos grupos se ha mantenido la configuración por defecto que permite **todo el tráfico de salida** (`0.0.0.0/0`). Esto asegura que las instancias puedan comunicarse con servicios externos de AWS, descargar actualizaciones de seguridad y gestionar dependencias de software sin restricciones.

### 3. Errores Encontrados y Soluciones Aplicadas

|Error Detectado|Causa Raíz|Solución Técnica Aplicada|

### 4. Justificación de Seguridad (El "Por qué")

- **¿Por qué el Side-loading de paquetes?** Para garantizar que la base de datos nunca necesite un Gateway de Internet, reduciendo la superficie de ataque casi a cero.
    
- **¿Por qué la tabla `logs_seguridad`?** Para centralizar la telemetría de ataques detectados por los scripts PHP (Honeypot), permitiendo análisis forense posterior.
    
- **¿Por qué `adminweb@'%'` limitado por IP de red?** Aunque el usuario acepta conexiones externas, el firewall de AWS (Security Group) solo permite que esa IP sea la de nuestro servidor web.

## 🛡️ Resumen Técnico de la Implementación de Seguridad (Wazuh SIEM)

### 1. Arquitectura de Despliegue

Se ha implementado una solución **SIEM (Security Information and Event Management)** basada en Wazuh, distribuida en una arquitectura de tres capas dentro de una VPC de AWS:

- **Wazuh Manager (Cerebro):** Desplegado en una instancia **t3.medium** con Ubuntu 24.04. Centraliza el análisis de logs, la gestión de alertas y el almacenamiento de eventos de seguridad.
    
- **Wazuh Agents (Vigilantes):** Instalados en los nodos críticos:
    
    - **Servidor Web (10.0.1.250):** Amazon Linux 2023.
        
    - **Servidor de Base de Datos (10.0.2.61):** Amazon Linux 2023 (Subred privada).
        

---

### 2. Hitos de Configuración Técnica

#### A. Gestión de Memoria en Instancias Limitadas

Para asegurar la estabilidad del Manager, se configuró un área de **memoria de intercambio (Swap)** de 2GB. Esto previene fallos del motor de indexación (Indexer) durante picos de análisis de tráfico.

#### B. Despliegue en Entornos Aislados (Bastion Host Strategy)

Debido a que la Base de Datos reside en una subred privada sin acceso a internet, se implementó una estrategia de **pasarela interna**:

1. Uso del Manager como repositorio temporal de paquetes vía HTTP (Puerto 80).
    
2. Transferencia interna de binarios `.rpm` evitando la exposición de llaves privadas (SSH Keys) entre servidores.
    

---

### 3. Módulos de Seguridad Activados

|**Módulo**|**Implementación**|**Valor para el Proyecto**|
|---|---|---|
|**FIM (File Integrity Monitoring)**|Vigilancia en tiempo real de `/var/www/html` y `/etc/mysql`.|Detecta instantáneamente defacements (cambios en la web) o alteraciones en la configuración de la BD.|
|**Vulnerability Detector**|Escaneo periódico contra bases de datos de CVE.|Identifica software desactualizado o crítico en el inventario de AWS.|
|**Log Data Analysis**|Análisis de logs de sistema (SSH, Syslog).|Monitorización de intentos de acceso no autorizados y ataques de fuerza bruta.|
|**SCA (Security Configuration Assessment)**|Escaneo de endurecimiento (Hardening).|Verifica si el sistema operativo cumple con las mejores prácticas de seguridad (benchmarks de CIS).|

---

### 4. Flujo de Comunicación y Seguridad de Red

Se han securizado las comunicaciones mediante **Security Groups** de AWS, permitiendo tráfico exclusivamente por los puertos necesarios:

- **TCP 1514/1515:** Comunicación cifrada entre agentes y manager.
    
- **TCP 443:** Acceso seguro al Dashboard de administración.
    
- **Aislamiento:** El puerto 80 se utilizó de forma efímera para el despliegue y se cerró posteriormente para seguir el principio de **Mínimo Privilegio**.
